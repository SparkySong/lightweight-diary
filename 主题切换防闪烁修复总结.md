# 微信小程序主题切换防闪烁修复总结

## 一、问题描述

在小程序中手动切换主题（浅色 / 深色 / 跟随系统）后：

- **切 tab 瞬间，目标页面整个背景会从白色闪烁为黑色（或反之）**
- 闪烁仅在**主题切换后第一次切 tab** 时出现
- 模拟器、iPhone 真机均可复现

尝试过但**未彻底解决**的常见方案：

- `page-meta` 组件的 `background-color` / `root-background-color` 动态绑定
- `wx.setNavigationBarColor` 设置 `duration: 0`
- `notifyThemeChange` 遍历页面栈同步 `setData`
- `theme.json` + `darkmode: true` + `@变量` 机制
- 全屏固定背景遮罩层 `theme-bg-fixed`（`position: fixed; z-index: -1`）
- CSS `transition: background-color` 平滑过渡
- `visibility: hidden` 临时隐藏页面 + 遮罩兜底

---

## 二、根因分析（两层叠加）

### 1. `app.json` 的 `@变量` 跟随的是**微信客户端主题**，不是 app 内手动主题

```json
"backgroundColor": "@backgroundColor"
```

`@backgroundColor` 解析的是**微信客户端当前主题**对应的 `theme.json` 字段：

- 微信客户端浅色 → `@backgroundColor` = `#F8FAF9`（白）
- 微信客户端深色 → `@backgroundColor` = `#121212`（黑）

用户在 app 内手动切到深色主题，但微信客户端仍是浅色 → `app.json` 的底色**依然是白色** → 切 tab 时框架先渲染白色底 → `.theme-dark` CSS 覆盖为黑 → **"白→黑"闪烁**。

### 2. 隐藏 tab 页面的 WebView 被冻结，`reLaunch` 重建瞬间有系统壳层过渡帧

- 每个 tab 页面是**独立的 WebView**
- 隐藏期间 WebView 被**暂停渲染**（微信性能优化）
- 切回时先显示冻结帧的像素快照，`setData`、`visibility: hidden` 等 JS 层方案在**第 1-2 帧内根本来不及介入**
- `wx.reLaunch` 销毁重建 WebView 的瞬间，屏幕会短暂露出**微信客户端壳层色**

---

## 三、最终方案：硬编码底色 + `reLaunch` 重置 + 双阶段 loading 接力

### 核心思路

1. 让**任何情况下** `app.json` 底色都是深色（硬编码 `#121212`），消除跟随系统造成的白色底
2. 手动切主题时通过 `wx.reLaunch` **彻底销毁重建所有 tab 的 WebView**，所有页面用新主题从零初始化，不存在"旧 DOM 残留"
3. `reLaunch` 瞬间用 `wx.showLoading({ mask: true })` 遮罩 + 新页面 `onLoad` **再开一次 loading 接力**，覆盖系统壳层过渡帧

### 流程时序

```
用户点击切主题
  │
  ├─▶ setStorageSync('appTheme', ...)                 保存主题
  ├─▶ app.globalData.theme = ...                      更新全局状态
  ├─▶ app.applyThemeToTabBar()                        更新 tabBar
  ├─▶ app.notifyThemeChange()                         通知当前页面
  ├─▶ setStorageSync('pendingThemeToast', msg)        记录待显示提示
  ├─▶ wx.showLoading({ title: '切换中...', mask: true })  全屏 loading
  │
  └─▶ setTimeout(250ms)
         │
         └─▶ wx.reLaunch({ url: '/pages/<当前tab>/...' })
                │
                ├─▶ 所有 tab 的 WebView 被销毁
                ├─▶ 新 WebView 按 app.json 深色底色启动
                │
                └─▶ 新页面 onLoad
                       │
                       ├─▶ 发现 pendingThemeToast 存在
                       ├─▶ 立即 wx.showLoading 接力覆盖
                       ├─▶ setTimeout(250ms) → hideLoading
                       │
                       └─▶ onShow
                              │
                              ├─▶ 读取 pendingThemeToast 并 showToast
                              └─▶ removeStorageSync('pendingThemeToast')
```

## 四、涉及文件与改动

### 4.1 `miniprogram/app.json`

**改动**：`window` 和 `tabBar` 的所有 `@变量` 引用改为硬编码深色值。

```json
{
  "darkmode": true,
  "themeLocation": "theme.json",
  "window": {
    "navigationBarBackgroundColor": "#121212",
    "navigationBarTextStyle": "white",
    "backgroundColor": "#121212",
    "navigationStyle": "custom",
    "enablePullDownRefresh": true,
    "backgroundTextStyle": "light"
  },
  "tabBar": {
    "color": "#6B7280",
    "selectedColor": "#5FA895",
    "backgroundColor": "#0f0f13",
    "borderStyle": "black"
  }
}
```

> **为什么不保留 `@变量`**：`@变量` 跟随微信客户端主题，无法跟随 app 内手动主题，在"微信浅色 + app 深色"场景下必然导致白色底闪烁。硬编码深色后，任何场景下底色都是深色，配合 `page` 默认深色 CSS，不会再看到白底。

### 4.2 `miniprogram/app.js`

`notifyThemeChange` 在切主题时同步：
- 记录时间戳 `globalData.themeChangedAt`（可选，用于其他场景判断）
- 对所有已加载页面 `setData({ currentTheme, themeSetting })`
- `hidePage` 预设（兼容保留，reLaunch 后实际不会触发）

### 4.3 `miniprogram/pages/profile/profile.js` 与 `miniprogram/pages/index/index.js`

**`applyThemeWithSystem` 末尾改动**：

```js
// 记录切换后的提示文案，reLaunch 后由新页面读取并显示
let toastMsg = '';
if (themeSetting === 'system') toastMsg = '已切换到跟随系统';
else if (themeSetting === 'dark') toastMsg = '切换到深色模式';
else toastMsg = '切换到浅色模式';
wx.setStorageSync('pendingThemeToast', toastMsg);

// 显示 loading 遮罩 → reLaunch 到当前 tab，彻底重置所有 tab 的 WebView
wx.showLoading({ title: '切换中...', mask: true });
setTimeout(() => {
  wx.reLaunch({
    url: '/pages/profile/profile',  // index.js 里改为 '/pages/index/index'
    complete: () => {
      wx.hideLoading();
    }
  });
}, 250);
```

### 4.4 `miniprogram/pages/{index,diet,profile}/*.js` —— 三个 tab 页面 `onLoad` 统一改动

**`onLoad` 开头接力 loading**：

```js
onLoad() {
  // 接力主题切换的 loading 遮罩，覆盖 reLaunch 瓦解瞬间的系统壳层过渡帧
  if (wx.getStorageSync('pendingThemeToast')) {
    wx.showLoading({ title: '切换中...', mask: true });
    setTimeout(() => wx.hideLoading(), 250);
  }
  // ...原有初始化逻辑
}
```

**`onShow` 调用 `showPendingThemeToast`**：

```js
onShow() {
  this.initTheme();
  this.showPendingThemeToast();  // 显示 reLaunch 后的主题切换提示
  // ...原有逻辑
}

// 读取 reLaunch 前存入的主题切换提示并显示一次
showPendingThemeToast() {
  const msg = wx.getStorageSync('pendingThemeToast');
  if (!msg) return;
  wx.removeStorageSync('pendingThemeToast');
  this.showToast(msg);
}
```

### 4.5 `miniprogram/app.wxss`

保留 `page` 默认深色 + `transition` + `theme-bg-fixed` 全屏固定遮罩层：

```css
page {
  color: #F5F5F5;
  background-color: #121212;
  transition: background-color 0.25s ease, color 0.25s ease;
}
page.theme-light {
  color: #1F2937;
  background-color: #F8FAF9;
}
.theme-bg-fixed {
  position: fixed;
  top: 0; left: 0; right: 0; bottom: 0;
  z-index: -1;
  background-color: #121212;
  transition: background-color 0.25s ease;
  pointer-events: none;
}
.theme-bg-fixed.theme-light {
  background-color: #F8FAF9;
}
```

### 4.6 `miniprogram/theme.json`

`light` 和 `dark` 都补上 `backgroundTextStyle` 字段（`light → "dark"`、`dark → "light"`），修复 `@backgroundTextStyle` 引用时报 "not found" 的编译错误。

> 注：由于 `app.json` 已改为硬编码，`theme.json` 实际在本方案中不再生效，但保留配置避免 darkmode 机制冲突。

## 五、效果

| 场景 | 修复前 | 修复后 |
|---|---|---|
| 微信深色 + app 深色 | 无闪（本就无问题） | 无闪 ✅ |
| 微信深色 + app 浅色 | 略有闪烁 | **无闪** ✅ |
| 微信浅色 + app 浅色 | 无闪（本就无问题） | 无闪 ✅ |
| 微信浅色 + app 深色 | **严重白→黑闪烁** | **模拟器无闪、真机仅 1-2 帧极快速闪烁** ✅ |

整个主题切换过程 ≈ **500ms**（250ms 旧页面 loading + reLaunch + 250ms 新页面 loading）。用户感知是"切换中... → 切换完成 + toast"，连续 loading 体验自然。

## 六、已知限制

1. **iPhone 真机"微信浅色 + app 深色"场景仍有 1-2 帧极快速闪烁**  
   根因：iOS WebKit 销毁/创建 WebView 的硬件合成过渡是系统级行为，JS 层无法介入。这是微信框架底层限制，**app 侧已无优化空间**。

2. **切主题会重置页面栈**  
   `wx.reLaunch` 会销毁页面栈。由于切主题入口在 tab 页面，本身不需要保留栈，可忽略。

3. **切换期间 500ms 不可交互**  
   loading 遮罩期间用户无法操作。属于可接受体验。

4. **`theme.json` / `@变量` 机制未使用**  
   因为 `@变量` 无法跟随 app 内手动主题，本方案改用 `app.json` 硬编码 + CSS 切换。如果产品后续改为"纯跟随系统主题"，可以把 `@变量` 恢复并去掉 `reLaunch` 逻辑。

## 八、经验沉淀

1. **`app.json` 的 `@变量` 不是"跟随 app 主题"**，是"跟随微信客户端主题"。真正需要 app 独立主题时必须硬编码。
2. **隐藏 tab 页面的 WebView 会冻结渲染**，任何基于 `setData` 的跨 tab 同步都有延迟窗口。
3. **`wx.reLaunch` 是解决"已加载 tab 状态不一致"的核武器**，但会瞬间露出系统壳层，需要 loading mask 遮盖。
4. **`wx.reLaunch` 会自动关闭 `showLoading`**，所以需要新页面 `onLoad` 里再次 `showLoading` 接力。
5. **所有"切换后提示"应通过 storage 传递**（如 `pendingThemeToast`），不能依赖页面实例状态（reLaunch 后实例全被销毁）。
